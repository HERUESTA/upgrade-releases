# This Week in Rails コードリーディング 手順書

毎週金曜日ごろに公開される "This Week in Rails" 記事を使って、
Railsのコード変更を読む習慣をつけるための手順書。

## ゴール

- Railsの内部実装に少しずつ慣れる
- 「PRを読む型」を体得して、毎週継続できるようにする
- 全件読まなくていい。1〜2件を深く読む方が価値がある

---

## ステップ1: 記事を流し読む

- 記事URL: `https://rubyonrails.org/blog` から最新の "This Week in Rails" を開く
- 全PRのタイトルと1行説明を読む
- カテゴリをざっくり把握する

| カテゴリ | 例 |
|---------|---|
| 機能追加 | 新しいオプション、新しいメソッド |
| バグ修正 | 予期しない動作の修正 |
| パフォーマンス | 速度改善、メモリ削減 |
| 非推奨化 | `deprecated` 警告の追加、削除 |
| DB対応 | MySQL/PostgreSQL固有の対応 |
| DevExp | Dockerfile、ジェネレータ、設定まわり |
| その他 | ドキュメント、CI |

---

## ステップ2: 深掘りするPRを1〜2件選ぶ

全件読む必要はない。以下の基準で選ぶ：

- **好奇心**: タイトルだけでは何をしているか分からない → 読む価値あり
- **身近さ**: 自分がよく使う機能（ActiveRecord, ActionText, etc.）に関係する
- **読みやすさ**: 変更行数が少ない（GitHubで `+10 -5` くらいのもの）
- **学習効率**: 初めて見る概念・APIが含まれている

選んだPRに ★ をつけてPR一覧テーブルに記録する。

---

## ステップ3: PRの背景を読む

1. GitHubのPRページを開く（記事のリンクから）
2. **最初のコメント（PR description）** を読む
   - 何が問題だったか
   - 何を解決するか
3. 長い場合は最初の2〜3段落で十分
4. メモに「背景」として書き留める

---

## ステップ4: テストdiffを読む

1. "Files changed" タブを開く
2. テストファイル（`*_test.rb` や `spec/**/*.rb`）を探す
3. `+` の行（追加行）だけに集中して読む
4. `assert_*` や `expect(...)` の内容から「何を期待しているか」を把握する

**ポイント**: `-` の行（削除行）は今は無視してOK。追加テストが「新しい仕様」を示している。

---

## ステップ5: 実装diffを読む

1. テストで把握した「期待動作」を前提に、実装ファイルのdiffを読む
2. 変更の核心（追加された1〜3行）を特定する
3. メソッド名や変数名から「方針」を掴む

**変更が小さいとき（〜20行）**: 全部読んで「何をしたか」を一言でまとめる
**変更が大きいとき（20行〜）**: 関数名・構造から「どういうアプローチか」を掴む

---

## ステップ6: 振り返り

- このPRで **何を学んだか** を1文で書く
- 知らなかった用語・APIがあれば 📝 用語メモに追加する

---

## ファイル管理

```
docs/Rails/
├── step.md              ← この手順書
├── template/
│   └── my.md            ← 毎週コピーして使うテンプレート
└── YYYY_MM_DD.md        ← 各週のノート（日付はこの-week-in-railsの公開日）
```

新しい週を始めるとき:
```bash
cp docs/Rails/template/my.md docs/Rails/YYYY_MM_DD.md
```
